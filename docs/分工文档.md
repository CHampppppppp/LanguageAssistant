# 项目分工说明

这份文档记录当前阶段三位同学的后续分工。项目已经完成了一些基础整理，所以本文按“当前剩余工作”重新划分任务，以最终可交付和可验收为准。

## 分工原则

- 按模块边界分工，避免三个人同时改同一条主流程。
- 吴负责页面整合和最终集成，因此承担略多的收口工作。
- 分工按后续剩余工作量估算，不按总代码量估算。
- 每个人都要有可验收的交付物，不能只写“协助”。
- 涉及密钥、`.env`、删除数据集、删除目录、git 历史等操作，必须先确认。

## 当前已完成的基础更新

这些事情已经完成，后续分工里不再重复当成主要工作量：

- 已新增 `AGENTS.md`，作为项目协作和工程规则。
- 已新增 `docs/项目介绍.md`，说明鸿蒙项目目录结构。
- 已新增并更新 `docs/分工文档.md`。
- 已新增 `.env.example`，只放环境变量占位符，不放真实密钥。
- `.gitignore` 已忽略 `.env` 和 LibriSpeech 数据集目录。
- `entry/src/main/ets/config/IFlytekConfig.ets` 已移除硬编码密钥，改成空配置和缺失配置校验。
- LibriSpeech 数据集约定为本地资源，不跟 git 上传。

后续重点不是重复建文档，而是把 API、识别、评测、语义理解、指令执行和页面流程真正补到可演示状态。

## 人员角色

当前三位同学的角色如下：

- 潘：项目结构、反馈、测试规范、科大讯飞 API 服务边界负责人。
- 胡：录音采集、语音识别链路、LibriSpeech、准确率/响应时间测试负责人。
- 吴：语义理解、Transformer 实验模块、指令执行、页面整合、最终集成负责人。

| 人员 | 后续占比 | 核心职责 |
|---|---:|---|
| 潘 | 33% | 项目结构、反馈模块、测试规范、科大讯飞 API 服务边界 |
| 胡 | 33% | 录音采集、语音识别链路、LibriSpeech、准确率/响应时间测试 |

## 潘：项目结构、反馈、测试规范、科大讯飞 API 服务边界

### 定位

潘负责项目规则、反馈能力、测试规范和科大讯飞 API 接入边界。潘的任务偏“基础设施”和“质量保障”，要保证项目后续开发不会继续出现密钥硬编码、测试空壳、反馈占位不清楚的问题。

### 任务

1. 维护项目结构和文档
   - 维护 `AGENTS.md`。
   - 维护 `docs/项目介绍.md`。
   - 维护 `docs/分工文档.md`。
   - 确保新增文件和目录符合项目约定。

2. 科大讯飞 API 接入
   - 在当前“空配置 + 校验”的基础上，完成可运行的安全配置读取方案。
   - 保持 `appId`、`apiKey`、`apiSecret` 不写死进源码。
   - 保持缺少配置时给出明确错误提示。
   - 整理 `core/asr/` 下的 API 请求参数、签名、响应解析和错误码处理。
   - 签名失败必须明确失败，不能用不等价的 SHA256 备用签名假装成功。
   - 与胡对齐 `AsrService` 接口，但不负责麦克风采集、录音状态机和 LibriSpeech 评测。

3. 反馈模块整理
   - 整理 `VoiceFeedback`。
   - 如果来不及接 TTS，明确称为文字反馈或模拟语音反馈。
   - 反馈内容要和指令执行结果一致。
   - 错误情况也要给用户可理解的反馈。

4. 测试体系
   - 替换当前模板测试。
   - 制定核心模块测试边界，推动每个负责人补自己模块的测试。
   - 补 ASR 响应解析、错误码和缺失配置测试，真实 API 调用要 mock。
   - 配合吴清理 `SemanticUnderstanding`、`CommandExecutor` 的模板测试。
   - 配合胡清理 `PerformanceMonitor` 和评测相关模板测试。

### 交付物

- 项目结构和文档保持一致。
- 科大讯飞 API 可以读取安全配置并完成请求构造。
- 反馈模块边界清楚。
- 测试规范明确，ASR 相关模板测试被替换为有效业务测试。

## 胡：录音采集、语音识别、LibriSpeech、准确率/响应时间测试

### 定位

胡负责录音采集、语音识别主链路和项目最硬的验收指标：语音转写准确率不低于 93%，系统响应时间不超过 2 秒。这里不能只做 UI 展示，必须能解释数据怎么来的。

### 任务

1. 语音识别链路
   - 整理 `SpeechRecognizer`，对接潘提供的 `AsrService`。
   - 负责 `core/audio/` 下的麦克风采集、录音状态机和音频数据处理。
   - 保证音频格式、采样率、声道和讯飞 API 输入要求匹配。
   - 明确中文实时交互和 LibriSpeech 英文评测的区别。

2. LibriSpeech 数据集处理
   - 确认数据集本地路径：`entry/src/main/resources/rawfile/dev-clean/LibriSpeech/`。
   - 保持数据集不进 git。
   - 加载 `.trans.txt` 标准文本。
   - 处理音频读取和解码问题，不能把未解码的 FLAC 当 PCM 用。

3. 准确率和响应时间测试
   - 设计准确率计算方式，优先使用 WER 或 word accuracy。
   - 统计平均响应时间。
   - 输出评测结果：样本数、准确率、平均响应时间、是否达标。
   - 评测结果不能用 `command.confidence * 100` 冒充语音转写准确率。

4. 与潘、吴的接口协作
   - 与潘对齐讯飞 API 服务接口。
   - 与吴对齐识别文本输出格式，保证语义理解能稳定消费。
   - 不直接修改页面业务逻辑和指令执行规则，页面接入由吴收口。

### 交付物

- 可用的语音识别链路。
- 录音状态机稳定，能处理开始、停止、失败和空音频场景。
- LibriSpeech 数据集评测入口。
- 准确率和响应时间统计结果。
- 报告中能说明 93% 准确率如何计算。

## 吴：语义理解、Transformer 实验模块、指令执行、页面整合、最终集成

### 定位

吴负责让系统像一个完整的语音助手，而不是只会把语音变成文字。重点是语义理解、Transformer 实验模块、指令执行、页面整合和最终演示版本集成。

### 任务

1. 语义理解模块完善
   - 整理意图枚举或常量，避免到处散落字符串。
   - 完善关键词和正则规则。
   - 支持基础指令：时间、天气、音乐、音量、闹钟、打开应用。
   - 补实体抽取，例如时间、应用名、歌曲名。

2. Transformer 实验模块整理
   - 将 Transformer 定位为 ASR 转写后的语义理解实验模块，不宣称替代讯飞 ASR。
   - 明确输入为识别文本，输出为意图分类、语义特征或重排序结果。
   - 如果暂时不接入主流程，必须在文档和 UI 文案中说明它是实验模块。
   - 保留可运行入口或最小演示，避免只有类名和空 `forward()`。

3. 指令执行模块完善
   - 时间查询使用真实本地时间。
   - 天气、音乐、音量、闹钟、打开应用如果暂时无法真实调用，要明确标记为模拟执行。
   - 每个执行结果返回状态：成功、失败、模拟、错误原因。

4. 流程编排
   - 在 `VoiceAssistantViewModel` 中串联胡的录音/识别结果、吴的语义理解/指令执行和潘的反馈模块。
   - 页面层只消费明确状态，底层录音状态机由胡负责。
   - 区分录音中、识别中、执行中、失败、完成等页面展示状态。

5. 页面整合和最终集成
   - 整理 `entry/src/main/ets/pages/Index.ets` 的页面状态。
   - 把潘的 API 和反馈模块接入主流程。
   - 把胡的语音识别和数据集评测结果接回页面。
   - 确保最终演示流程可以完整跑通。
   - UI 文案不能夸大未实现能力，例如 Transformer 未真正接入时不能宣传成已完成。

### 交付物

- 至少 6 类基础指令可识别：时间、天气、音乐、音量、闹钟、打开应用。
- Transformer 模块定位清楚，有实验入口或明确的未接入说明。
- 指令执行结果稳定。
- 录音到反馈的完整流程稳定。
- 页面展示和最终演示版本完成集成。

## 个人完成标准与验收方式

分工按负责人划分，验收按功能模块划分。判断一个人的工作是否完成时，不只看代码是否写了，而是看模块能否通过对应验证，并能提供测试、截图、日志或评测结果作为证据。

| 模块 | 负责人 | 完成标准 | 验证方式 |
|---|---|---|---|
| 科大讯飞 ASR 服务 | 潘 | 无硬编码密钥；缺少配置时有明确错误；签名、请求构造、响应解析和错误码处理封装在 `core/asr/` | ASR mock 测试通过；源码中搜不到真实 `appId`、`apiKey`、`apiSecret` |
| 反馈模块 | 潘 | 成功、失败、模拟执行、未知指令都有对应反馈；未接入 TTS 时不宣传为真实语音播报 | 手动触发成功、失败、未知指令场景，页面反馈文案准确 |
| 测试规范 | 潘 | 模板测试被识别并替换；每个核心模块有对应测试责任人 | `entry/src/test/` 中不再只有无业务意义的模板测试 |
| 录音采集 | 胡 | 开始、停止、失败、空音频状态稳定；录音状态机归 `core/audio/` 管理 | 真机或模拟器验证开始录音、停止录音、拒绝权限、空音频场景 |
| 语音识别链路 | 胡 | `SpeechRecognizer` 能对接潘的 `AsrService`，输出稳定识别文本或明确错误 | 手动完成录音到识别文本流程；网络/API 失败时有错误结果 |
| LibriSpeech 评测 | 胡 | 能加载 `.trans.txt`；不把 FLAC 当 PCM；能输出样本数、准确率、平均响应时间和是否达标 | 运行评测入口并保存评测结果，报告能解释 93% 和 2 秒指标 |
| 性能统计 | 胡 | 响应时间口径明确；准确率来自 WER 或 word accuracy，不使用置信度冒充 | 评测结果包含 response time、WER/word accuracy、样本数 |
| 语义理解 | 吴 | 至少 6 类基础指令可识别；未知指令不误判为高置信度；实体抽取结果稳定 | `SemanticUnderstanding` 测试覆盖正常、未知、边界输入 |
| Transformer 实验模块 | 吴 | 明确定位为 ASR 转写后的语义理解实验模块；有最小演示入口或明确未接入说明 | 运行实验入口，或在文档/UI 中明确说明未接入主流程 |
| 指令执行 | 吴 | 每类指令返回成功、失败、模拟或错误原因；模拟能力不伪装成真实系统调用 | `CommandExecutor` 测试覆盖成功、失败、模拟场景 |
| 页面与流程集成 | 吴 | 页面能串联录音、识别、语义理解、指令执行和反馈；状态展示不混乱 | 手动跑完整链路：开始录音 -> 停止录音 -> 返回识别文本 -> 执行指令 -> 展示反馈 |

每个人交付时至少提供：

- 改动的文件或目录。
- 已完成的模块清单。
- 已运行的测试或手动验证步骤。
- 未完成或仍是模拟能力的说明。
- 对其他人的接口依赖，例如 `AsrService` 输入输出、识别文本格式、指令执行结果格式。
