# 项目分工说明

这份文档记录当前阶段三位同学的后续分工。前提是：原始项目框架已经完成，后续任务按模块归属重新划分，以最终可交付和可验收为准。

## 分工原则

- 按模块边界分工，避免三个人同时改同一条主流程。
- C 负责页面整合和最终集成，因此承担略多的收口工作。
- 分工按后续剩余工作量估算，不按总代码量估算。
- 每个人都要有可验收的交付物，不能只写“协助”。
- 涉及密钥、`.env`、删除数据集、删除目录、git 历史等操作，必须先确认。

## 人员角色

为了方便描述，本文先用 A、B、C 代称：

- A：项目结构、反馈、测试、科大讯飞 API 负责人。
- B：语音识别、LibriSpeech、准确率/响应时间测试负责人。
- C：语义理解、指令执行、流程修补、页面整合、最终集成负责人。

如果你们要换成真实姓名，直接替换 A、B、C 即可。

## 工作量比例

| 人员 | 后续占比 | 核心职责 |
|---|---:|---|
| A | 33% | 项目结构、反馈模块、测试、科大讯飞 API |
| B | 33% | 语音识别、LibriSpeech、准确率/响应时间测试 |
| C | 34% | 语义理解、指令执行、流程修补、页面整合、最终集成 |

说明：C 负责最终集成和页面收口，后期协调成本会更高，所以比例略高一点。A、B、C 的实际工作量仍以交付物完成为准。

## A：项目结构、反馈、测试、科大讯飞 API

### 定位

A 负责项目规则、反馈能力、测试体系和科大讯飞 API 接入边界。A 的任务偏“基础设施”和“质量保障”，要保证项目后续开发不会继续出现密钥硬编码、测试空壳、反馈占位不清楚的问题。

### 任务

1. 维护项目结构和文档
   - 维护 `AGENTS.md`。
   - 维护 `PROJECT_STRUCTURE.md`。
   - 维护 `WORK_DIVISION.md`。
   - 确保新增文件和目录符合项目约定。

2. 科大讯飞 API 接入
   - 根据 `.env.example` 设计安全配置读取方案。
   - 不允许把 `appId`、`apiKey`、`apiSecret` 写死进源码。
   - 缺少配置时给出明确错误提示。
   - 整理 API 请求参数、签名、响应解析和错误码处理。
   - 签名失败必须明确失败，不能用不等价的 SHA256 备用签名假装成功。

3. 反馈模块整理
   - 整理 `VoiceFeedback`。
   - 如果来不及接 TTS，明确称为文字反馈或模拟语音反馈。
   - 反馈内容要和指令执行结果一致。
   - 错误情况也要给用户可理解的反馈。

4. 测试体系
   - 替换当前模板测试。
   - 补 `SemanticUnderstanding` 测试。
   - 补 `CommandExecutor` 测试。
   - 补 `PerformanceMonitor` 测试。
   - 配合 B 补 ASR 响应解析和错误码测试，真实 API 调用要 mock。

### 交付物

- 项目结构和文档保持一致。
- 科大讯飞 API 不再硬编码密钥。
- 反馈模块边界清楚。
- 模板测试被替换为有效业务测试。

## B：语音识别、LibriSpeech、准确率/响应时间测试

### 定位

B 负责语音识别主能力和项目最硬的验收指标：语音转写准确率不低于 93%，系统响应时间不超过 2 秒。这里不能只做 UI 展示，必须能解释数据怎么来的。

### 任务

1. 语音识别链路
   - 整理 `SpeechRecognizer` 或拆出独立 ASR 服务。
   - 负责麦克风采集到音频数据的处理。
   - 保证音频格式、采样率、声道和讯飞 API 输入要求匹配。
   - 明确中文实时交互和 LibriSpeech 英文评测的区别。

2. LibriSpeech 数据集处理
   - 确认数据集本地路径：`entry/src/main/resources/rawfile/dev-clean/LibriSpeech/`。
   - 保证数据集不进 git。
   - 加载 `.trans.txt` 标准文本。
   - 处理音频读取和解码问题，不能把未解码的 FLAC 当 PCM 用。

3. 准确率和响应时间测试
   - 设计准确率计算方式，优先使用 WER 或 word accuracy。
   - 统计平均响应时间。
   - 输出评测结果：样本数、准确率、平均响应时间、是否达标。
   - 评测结果不能用 `command.confidence * 100` 冒充语音转写准确率。

4. 与 A、C 的接口协作
   - 与 A 对齐讯飞 API 服务接口。
   - 与 C 对齐识别文本输出格式，保证语义理解能稳定消费。

### 交付物

- 可用的语音识别链路。
- LibriSpeech 数据集评测入口。
- 准确率和响应时间统计结果。
- 报告中能说明 93% 准确率如何计算。

## C：语义理解、指令执行、流程修补、页面整合、最终集成

### 定位

C 负责让系统像一个完整的语音助手，而不是只会把语音变成文字。重点是语义理解、指令执行、流程修补、页面整合和最终演示版本集成。

### 任务

1. 语义理解模块完善
   - 整理意图枚举或常量，避免到处散落字符串。
   - 完善关键词和正则规则。
   - 支持基础指令：时间、天气、音乐、音量、闹钟、打开应用。
   - 补实体抽取，例如时间、应用名、歌曲名。

2. 指令执行模块完善
   - 时间查询使用真实本地时间。
   - 天气、音乐、音量、闹钟、打开应用如果暂时无法真实调用，要明确标记为模拟执行。
   - 每个执行结果返回状态：成功、失败、模拟、错误原因。

3. 流程修补
   - 修正开始录音、停止录音、处理录音之间的状态冲突。
   - 区分录音中、识别中、执行中、失败、完成等状态。
   - 避免重复停止录音、空音频、设备释放后再次读取等问题。

4. 页面整合和最终集成
   - 整理 `entry/src/main/ets/pages/Index.ets` 的页面状态。
   - 把 A 的 API 和反馈模块接入主流程。
   - 把 B 的语音识别和数据集评测结果接回页面。
   - 确保最终演示流程可以完整跑通。
   - UI 文案不能夸大未实现能力，例如 Transformer 未真正接入时不能宣传成已完成。

### 交付物

- 至少 8 类基础指令可识别。
- 指令执行结果稳定。
- 录音到反馈的完整流程稳定。
- 页面展示和最终演示版本完成集成。

## 推荐开发顺序

1. A 先稳定项目结构、讯飞 API 配置边界和测试规则。
2. B 开始做语音识别链路、LibriSpeech 读取和指标测试。
3. C 并行做语义理解、指令执行和流程修补。
4. C 负责把 A、B 的模块接回页面并完成最终集成。
5. 三个人一起跑最终演示和报告数据。

## 最终验收清单

- App 能启动到首页。
- 麦克风录音流程能走通。
- 能返回语音转文字结果。
- 能识别基础语音指令。
- 能展示执行结果和反馈。
- LibriSpeech 数据集不进 git。
- 能输出准确率和响应时间。
- 报告能解释 93% 和 2 秒指标。
- 源码里没有真实密钥。
- 模板测试被替换为有效业务测试。
