# 项目分工说明

这份文档记录当前阶段三位同学的后续分工。前提是：原始项目框架已经由其中一位同学完成，所以后续任务不再三人完全平均，而是让原框架作者少承担一点新功能开发，更多负责结构收口和最终集成。

## 分工原则

- 原框架作者已经完成了项目雏形，后续工作量适当减少。
- 另外两位同学承担更多功能补齐、指标评测和测试任务。
- 分工按后续剩余工作量估算，不按总代码量估算。
- 每个人都要有可验收的交付物，不能只写“协助”。
- 涉及密钥、`.env`、删除数据集、删除目录、git 历史等操作，必须先确认。

## 人员角色

为了方便描述，本文先用 A、B、C 代称：

- A：原始框架作者。
- B：语音识别与数据集评测负责人。
- C：语义理解、指令执行与测试负责人。

如果你们要换成真实姓名，直接替换 A、B、C 即可。

## 工作量比例

| 人员 | 后续占比 | 核心职责 |
|---|---:|---|
| A 原框架作者 | 25% | 项目结构、页面整合、原有流程修补、最终集成 |
| B | 37.5% | 科大讯飞 API、语音识别、LibriSpeech、准确率/响应时间评测 |
| C | 37.5% | 语义理解、指令执行、反馈、业务测试 |

## A：原框架作者

### 定位

A 已经完成原始框架，所以后续不再承担最多的新功能开发。A 的重点是把现有框架整理清楚，帮助 B 和 C 的功能能稳定接入页面。

### 任务

1. 维护项目结构和文档
   - 维护 `AGENTS.md`。
   - 维护 `PROJECT_STRUCTURE.md`。
   - 确保新增文件和目录符合项目约定。

2. 页面和状态整理
   - 整理 `entry/src/main/ets/pages/Index.ets` 的页面状态。
   - 区分录音中、识别中、执行中、失败、完成等状态。
   - UI 文案不能夸大未实现能力，例如 Transformer 未真正接入时不能宣传成已完成。

3. 录音流程配合修复
   - 协助修正开始录音、停止录音、处理录音之间的状态冲突。
   - 避免重复停止录音、空音频、设备释放后再次读取等问题。

4. 最终集成
   - 把 B 的语音识别和数据集评测结果接回页面。
   - 把 C 的语义理解、指令执行和反馈结果接回页面。
   - 确保最终演示流程可以完整跑通。

### 交付物

- 页面入口稳定可用。
- 主流程从按钮点击到结果展示能跑通。
- 项目结构和文档保持一致。
- 最终演示版本完成集成。

## B：语音识别与数据集评测负责人

### 定位

B 负责项目最硬的验收指标：语音转写准确率不低于 93%，系统响应时间不超过 2 秒。这里不能只做 UI 展示，必须能解释数据怎么来的。

### 任务

1. 科大讯飞 API 配置接入
   - 根据 `.env.example` 设计安全配置读取方案。
   - 不允许把 `appId`、`apiKey`、`apiSecret` 写死进源码。
   - 缺少配置时给出明确错误提示。

2. 语音转文字链路
   - 整理 `SpeechRecognizer` 或拆出独立 ASR 服务。
   - 修正 API 请求、签名、响应解析和错误处理。
   - 明确中文实时交互和 LibriSpeech 英文评测的区别。

3. LibriSpeech 数据集处理
   - 确认数据集本地路径：`entry/src/main/resources/rawfile/dev-clean/LibriSpeech/`。
   - 保证数据集不进 git。
   - 加载 `.trans.txt` 标准文本。
   - 处理音频读取和解码问题，不能把未解码的 FLAC 当 PCM 用。

4. 准确率和响应时间评测
   - 设计准确率计算方式，优先使用 WER 或 word accuracy。
   - 统计平均响应时间。
   - 输出评测结果：样本数、准确率、平均响应时间、是否达标。

### 交付物

- 可用的语音转文字服务。
- LibriSpeech 数据集评测入口。
- 准确率和响应时间统计结果。
- 报告中能说明 93% 准确率如何计算。

## C：语义理解、指令执行与测试负责人

### 定位

C 负责让系统像一个语音助手，而不是只会把语音变成文字。重点是识别用户意图、提取参数、执行或模拟执行指令，并补上业务测试。

### 任务

1. 语义理解模块完善
   - 整理意图枚举或常量，避免到处散落字符串。
   - 完善关键词和正则规则。
   - 支持基础指令：时间、天气、音乐、音量、闹钟、打开应用。
   - 补实体抽取，例如时间、应用名、歌曲名。

2. 指令执行模块完善
   - 时间查询使用真实本地时间。
   - 天气、音乐、音量、闹钟、打开应用如果暂时无法真实调用，要明确标记为模拟执行。
   - 每个执行结果返回状态：成功、失败、模拟、错误原因。

3. 反馈模块整理
   - 如果来不及接 TTS，明确称为文字反馈或模拟语音反馈。
   - 反馈内容要和执行结果一致。
   - 错误情况也要给用户可理解的反馈。

4. 测试补齐
   - 补 `SemanticUnderstanding` 测试。
   - 补 `CommandExecutor` 测试。
   - 补 `PerformanceMonitor` 测试。
   - 替换当前模板测试。

### 交付物

- 至少 8 类基础指令可识别。
- 指令执行结果稳定。
- 反馈内容清楚。
- 业务单测覆盖核心规则。

## 推荐开发顺序

1. A 先稳定项目结构、页面入口和文档。
2. B 开始做 ASR 配置、讯飞 API 和 LibriSpeech 评测。
3. C 并行做语义理解、指令执行和测试。
4. A 把 B、C 的模块接回页面。
5. 三个人一起跑最终演示和报告数据。

## 最终验收清单

- App 能启动到首页。
- 麦克风录音流程能走通。
- 能返回语音转文字结果。
- 能识别基础语音指令。
- 能展示执行结果和反馈。
- LibriSpeech 数据集不进 git。
- 能输出准确率和响应时间。
- 报告能解释 93% 和 2 秒指标。
- 源码里没有真实密钥。
- 模板测试被替换为有效业务测试。
